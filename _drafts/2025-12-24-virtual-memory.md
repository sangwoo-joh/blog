프로그램은 필연적으로 메모리를 쓴다. 모든 현대 컴퓨터는 폰 노이만 아키텍쳐를 따르는데, 이에 따르면 계산은 CPU에서 이루어져야 하지만 프로그램과 데이터는 메모리에 저장되어 있다. 그래서 CPU가 계산을 하려면 얘네를 레지스터까지 가져와야 한다. 혹은 반대로, CPU가 계산한 결과를 현실 세계에 반영하려면 이걸 메모리에 써야하는데, 레지스터로부터 출발해서 메모리까지 갖고 가야 한다. 이 과정을 더 효율적으로 하기 위해서 이 사이에 수많은 보조적인 단계가 생기게 되었고, 그 결과 우리가 일반적으로 아는 메모리 계층구조(Memory Hierarchy)가 탄생하게 되었다.

```
CPU (register) -> L1 Cache -> L2 (-> L3) -> RAM -> SSD or HDD
```

 * **레지스터**: CPU 내부에 있어서 가장 속도가 빠르다. 데이터 뿐만 아니라 실행할 명령어(기계어)도 직접 처리할 수 있다. 대신 용량이 매우 작아서 수백 바이트 수준이다. 속도는 거의 1 싸이클 미만이다.
 * **L1 캐시**: CPU 코어에 내장되어 있고 캐시 중에서 가장 빠르다. 보통 명령어를 위한 캐시(Instruction Cache)와 데이터를 위한 캐시(Data Cache)로 분리되는 경우가 많다. 대충 32-128KB 정도의 크기를 가진다. 속도는 러프하게 4 싸이클 정도라고 보면 된다.
 * **L2 캐시**: L1보다 용량은 크지만 좀 느린 캐시다. 역시 CPU 코어 근처에 있다. 용량은 256K-1MB 정도이고 속도는 10-20 싸이클 정도.
 * **L3 캐시**: 요건 선택사항이다. 멀티 코어 프로세서가 도입되면서 등장한 캐시로 여러 개의 코어가 공유하는 캐시다. 수십 MB 정도의 용량과 50-80 싸이클 정도의 속도를 가진다.
 * **RAM**: 여기부터는 이제 우리에게 익숙한 "메모리"다. RAM은 Random Access Memory의 줄임말이다. Random Access는 주소값만 알면 순차적인 접근 없이 데이터를 읽고 쓰고 할 수 있다는 뜻이다. 대충 요즘은 수, 수십, 수백 GB까지 다양하게 선택할 수 있다. 속도는 놀랍게도 100-300 싸이클 정도 된다. 프로그램이 데이터를 "쓰는(write)" 최종 목적지 중 하나다.
 * **저장장치**: SSD, HDD 등으로 전원이 꺼져도 데이터를 유지한다(비휘발성). 용량은 가격에 따라 수십, 수백기가에서 수테라까지 다양하다. 속도는 SSD냐 HDD냐에 따라서도 크게 다르지만 보통 RAM보다 수백, 수천, 수만배는 느리다. 프로그램이 데이터를 쓰는 최종 목적지 중 하나다.

아무튼 계산에 필요한 모든 코드와 데이터는 레지스터까지 와야 한다. 하지만 프로그램이 필요로 하는 데이터와 코드는 메모리에 있다. 캐시들은 레지스터와 메모리 사이에서 지역성(Locality)에 근거해서 프로그램이 더 빨리 실행되도록 돕는 보조적인 역할을 한다.

그런데 프로그램은 저마다 필요로 하는 메모리의 크기가 다르다. 어떤 애는 몇 메가만 있어도 되지만, 코드를 컴파일하거나 커다란 데이터를 분석하는 등의 작업에는 수십, 수백, 수테라의 메모리가 필요할 때도 있다. 하지만 물리적인 메모리(RAM)의 용량은 한계가 있다. 무엇보다 현대의 운영체제는 수많은 프로그램들을 동시에 실행해야 한다. 암만 메모리를 잘 쪼개서 프로그램마다 잘 할당한다고 해도, 어떤 프로그램이 얼만큼의 메모리가 필요한지를 항상 알아낼 수는 없기 때문에 (정지 문제), 이는 꽤 골치아프다.

모든 컴퓨터 과학 문제는 새로운 수준의 추상화를 추가해서 해결할 수 있다[^1]. **가상 메모리**는 이러한 문제를 해결해주는 추상화로, 모든 프로세스에게 자기만의 독립적인 가상 주소 공간을 제공한다. 실제 메모리는 가상-물리 메모리 매핑이 필요한 시점에 제공된다. 이를 통해 물리적으로는 한계가 있는 환경에서 프로그램에게는 (주소 공간이 허락하는 한) 거의 무한한 메모리 공간을 제공할 수 있다. 이 추상화 덕분에 커널은 진정한 의미의 동시성을 얻을 수 있었고, 더 나아가 메모리 보호(프로세스 간 가상 메모리에 접근 불가), 효율적인 메모리 공유, 그리고 스왑 공간을 통해 실제 필요한 메모리가 물리적인 메모리보다 더 큰 프로그램도 실행할 수 있는 등의 추가적인 이득도 얻었다.

현대의 가상 메모리 기법은 소프트웨어와 하드웨어 모두의 도움을 통해 구현된다. 소프트웨어 적으로는 페이징 기법이나 관련 알고리즘 등이 있고, 하드웨어 적으로는 MMU(Memory Management Unit; 가상 주소를 물리 주소로 변환하는 걸 도와줌)이나 TLB(Translation Lookaside Buffer; MMU를 위한 특수한 캐시) 등이 있겠다. 1950년대의 초기 컴퓨터에는 현대적인 의미의 동시성이라는 개념은 없었고 배치 처리(Batch Processing)를 통해서 여러 프로그램을 순차적으로 한 번에 하나씩 실행할 수 있었다. 그러다가 고정 파티션과 같은 제한적인 멀티 태스킹이 도입되기 시작했고, 역사에 따르면 여러 다양한 시도가 있었지만 각각 발목을 잡는 문제들, 예를 들면 심각한 단편화(Fragmentation)라던지 성능의 이슈라던지 하는 것들이 있었고, 이를 해결하기 위해서 수많은 연구자와 개발자들이 고군분투한 끝에 1970년대에 들어서는 지금의 우리가 아는 가상 메모리 시스템의 기본이 잡혔다.

아무튼간에 현대적인 컴퓨터에서는 프로그램이 보는 **모든** 메모리 주소는 가상 메모리 주소이다. 그리고 MMU 기반의 64비트 가상 메모리 시스템에서는, 메모리 주소는 일반적으로 다음과 같은 인코딩을 갖는다.
 * 페이지 번호 (VPN; Virtual Page Number): 상위 52비트. 이 번호를 가지고 페이지 테이블에서 페이지 테이블 엔트리 (PTE; Page Table Entry)를 찾고, 이걸 가지고 이 가상 메모리 페이지랑 매칭되는 물리 메모리 프레임 정보를 알아내어 메모리 연산을 할 수 있다.
 * 페이지 오프셋(Page Offset): 하위 12비트. 하나의 페이지가 관리하는 메모리 덩어리를 정의한다. 즉 한 페이지 안에서는 이 오프셋 범위 안에서 접근할 수 있다. 이 오프셋이 표현할 수 있는 크기가 곧 페이지(그리고 매칭되는 프레임)의 크기가 된다. 역사적으로 발견한 적당한 크기가 4KB라서 12비트를 주로 쓴다.

예를 들어, malloc으로 얻은 메모리 주소가 0x5555555552a0라면, 이중에서 상위 52비트인 0x555555555는 페이지 번호이고 하위 12비트인 0x2a0은 페이지 오프셋을 나타낸다. MMU는 이 인코딩을 해석해서, 먼저 TLB에 이 페이지 번호에 매칭되는 페이지 테이블 엔트리가 있으면 있으면 그걸 가져다 쓰고, 없으면 이제 페이지 테이블을 뒤져서 PTE를 찾은 다음 여기 든 정보를 이용해서 실제 물리 메모리의 주소 정보를 얻는다.

MMU는 이렇게 페이지 테이블 엔트리 정보를 이용해서 주소를 변환하고, 메모리 구역의 권한을 검사하고, 페이지 폴트가 발생하면 OS 인터럽트를 발생시키고, TLB를 관리하는 등의 역할을 보조한다. 이걸 커널 수준에서 구현해도 되지만 CPU 옆에 붙어 있는 MMU 덕분에 훨씬 더 효율적이고 빠르게 관리할 수 있다. 페이지 테이블 엔트리에는 다음과 같은 것들이 있다:
 * 페이지 번호와 매칭되는 물리 프레임 번호 (PFN). 참고로 오프셋은 동일한 값을 써서 프레임 번호만 알아내면 실제 물리 메모리 주소를 빠르게 계산할 수 있다.
 * 해당 물리 메모리에 대한 권한(쓰기, 읽기, 실행)
 * 유효 (Valid) 비트: 해당 페이지가 물리 메모리에 존재하는지 여부. 이게 0일 때 이 가상 페이지 엔트리에 접근하려고 하면 페이지 폴트가 발생한다. 그러면 커널이 스왑 영역에서 해당 페이지를 찾아서 페이지 정보를 RAM으로 로드하고 PFN을 부여한다.
 * Dirty 비트: 해당 페이지가 물리 메모리에 올라간 뒤에, 내용이 수정되었는지 여부. 0이면 변경된 적이 없고 1이면 변경된 적이 있다는 걸 뜻한다.
 * 모드: 사용자 모드인지 커널 모드인지를 구분한다.

이 페이지 테이블은 보통 커널에 의해서 관리되며 RAM의 한 구역을 차지하게 된다. 근데 64비트 주소 시스템에서 페이지 테이블 번호가 52비트 크기인데, 이걸 단순하게 할당하면 $$2^{52} \times 8B = 32PB$$ (여기서 8바이트는 PTE의 크기)라는 엄청나게 큰 메모리가 필요하게 되고 이러면 가상 메모리를 도입한 의미가 없어진다. 그래서 실제로는 52비트를 다 쓰지 않고, 추가로 Multi-Level Page Table, 그러니까 일종의 다단계를 도입해서 이 문제를 해결한다. 현재 x86-64에서는 4단계의 페이지 테이블을 이용한다고 하는데,
 * PML4 (Page Map Level 4): 항상 존재하는 최상위 테이블로 9비트, 즉 512개의 엔트리를 갖는다. 각각의 엔트리 크기는 8바이트로, 따라서 모든 페이지 테이블은 최소 $$ 512 \times 8 = 2^{11} = 4KB $$ 만큼의 용량은 항상 차지하게 된다.
 * 최상위 PML4 이후로 아래 3개의 하위 페이지들은 필요할 때에만 생성되며 각각 PDP(Page Directory Pointer), PD(Page Directory), PT(Page Table)로 불린다. 모두 9비트로 표현되며 각각 4KB의 크기를 갖는다.

이런 식으로 52비트 중 일부만 쓰고 나머지는 미래를 위해 예약해 둔 상황이다. 그래서 실제로는 64비트 시스템이더라도 48비트(9비트 4개 + 오프셋 12비트)만 사용하고 남은 16비트는 부호를 확장한 주소 체계를 쓰고 있다. 48비트만 쓰더라도 표현 가능한 가상 메모리 공간의 크기가 256TB씩이나 되어서 당분간은 걱정이 없다. 물리적인 제약이 완화되거나 더 큰 가상 메모리 공간이 필요하게 되면 페이지 테이블 단계를 하나 더 추가하거나 하면 된다. 실제로 인텔에서는 [5단계 페이지 테이블](https://en.wikipedia.org/wiki/Intel_5-level_paging)을 시범 도입해서 최대 128PB 메모리를 표현할 수 있는 57비트 주소 체계를 시도해보고 있다.

아무튼 이로 인해서 현대의 64비트 시스템의 Unix 기반 OS에서 돌아가는 프로그램은 독립적으로 256TB의 연속적인 가상 메모리 공간을 갖게 되었다. 그리고 보통 이걸 반으로 잘라서 128TB는 커널이 쓰고 나머지 절반을 사용자 프로그램이 쓸 수 있게 나눠놨다. 그래서 [데비안](https://www.debian.org/ports/amd64/) 문서를 보면 64비트 유저랜드에서는 프로세스마다 최대 128TiB의 가상 주소 공간을 갖게 된다는 설명이 있는 것이다.

그런데 이렇게 연속적으로 만든 가상 메모리 공간과 매칭되는 물리 메모리 공간은 제한적인 크기를 가지며 비연속적이다. 커널, BIOS, 펌웨어, 가상 메모리 시스템에 쓰이는 프로세스 별 페이지 테이블 등 많은 부수적인 정보들이 RAM에 함께 위치하기 때문에 제한적인 크기를 가지게 되고, 동시성을 제공하기 위해 수많은 프로그램이 동시에 메모리에 올라와서 실행되는데 이때 가상 메모리 주소와 물리 메모리 주소를 정말 딱 1:1로 짝짓지 않고 페이지와 프레임이라는 덩어리로 매칭하였기 때문에 페이지(프레임) 크기인 4KB 만큼 엇갈려서 (Interleaving) 배치되므로 비연속적이다. 이 4KB라는 페이지의 크기는 튜닝 가능한데, 페이지 크기가 커지면 한번에 커버하는 메모리 영역이 커지므로 대용량 데이터를 처리하는데 유리해지고 TLB 미스가 덜나지만, 내부 단편화가 증가하게 된다. 그러니까 우리가 아는 [지역성](https://en.wikipedia.org/wiki/Locality_of_reference), 즉 최근에 쓰인 데이터(메모리) 영역은 금방 다시 쓰인다는 점과 그 데이터와 인접한 데이터들도 함께 쓰인다는 관찰이 여기에 듬뿍 녹아들어 있는 것이다.

TLB 미스 하니까 생각났는데, TLB는 하드웨어 레벨 캐시로 히트한다면 1사이클 정도로 엄청나게 빠르다. 그리고 CPU 캐시랑 마찬가지로, 메모리에 저장된 것이 데이터일 수도 있고 코드일 수도 있어서, 이를 위해서 iTLB(for Instruction)과 dTLB(for Data)로 나뉘기도 한다. 그리고 TLB 미스가 발생하면 최대 100 사이클 정도의 미스 패널티가 발생한다. 이걸 기반으로 1000번의 메모리 접근이 이뤄질 때 대충 계산을 때려보면
 * TLB 히트 91%: 910번 히트 + 90번 미스 -> 총 9,910 사이클,
 * TLB 히트 90%: 900번 히트 + 100번 미스 -> 총 10,900 사이클,

대략 1%의 TLB 히트 레이트 차이가 10%의 성능 차이를 보여준다.

---

여기까지 가상 메모리에 대해서 아는 내용을 정리하며 떠들어 보았고, 요걸 바탕으로 생기는 다양한 엔지니어링 프랙티스에 대해서 좀더 얘기를 해볼까 한다.
